## Resolution

root_cause: |
  brainsuite_static_score.py defines its own BrainSuiteJobError class (line 46) that is a DIFFERENT Python class object from the BrainSuiteJobError imported into scoring_job.py from brainsuite_score.py (line 17). When the static scoring service raises its local BrainSuiteJobError during polling or submission, the except BrainSuiteJobError handler in scoring_job.py does NOT catch it (Python isinstance check fails on different class identity). The exception propagates to the bare except Exception block, which sets the error_reason string and calls _mark_failed(). If _mark_failed() then also fails (e.g., the DB session is in a bad state), its own try/except at line 226 swallows the error silently — leaving the row stuck at PENDING or in a state that gets observed as UNSCORED after a scheduler retry resets it.

fix: |
  Two options:
  A) PREFERRED: Extract all three exception classes (BrainSuiteRateLimitError, BrainSuite5xxError, BrainSuiteJobError) into a shared module (e.g. app/services/brainsuite_exceptions.py) and import from there in both service files and in scoring_job.py.
  B) QUICK FIX: In scoring_job.py, also import BrainSuiteJobError from brainsuite_static_score and catch both in the handler (or use a tuple catch).

  Going with Option A (shared exceptions module) as it is the correct fix and eliminates the root cause permanently.
