    **For each sync service**, find the location where the asset binary has been uploaded to MinIO (look for `obj_storage.upload_file(...)` or `storage.upload_file(...)` calls that set `asset.asset_url` or similar). Immediately AFTER the upload succeeds and the asset record is available with an `id` and `organization_id`, add:

    ```python
    import asyncio
    from app.services.ai_autofill import run_autofill_for_asset

    # After: served_url = obj_storage.upload_file(local_path, relative_path)
    # After: asset record is committed/flushed with an id
    asyncio.create_task(
        run_autofill_for_asset(asset_id=asset.id, org_id=asset.organization_id)
    )
    ```

    **Important implementation notes:**
    - The `import asyncio` should go at the top of the file (standard library section).
    - The `from app.services.ai_autofill import run_autofill_for_asset` should go in the local app imports section.
    - The `asyncio.create_task()` call must be AFTER the asset has been committed to the database (not just flushed) — otherwise the auto-fill service will not find the asset row. In each sync service, find the commit point after asset creation and place the call there.
    - If the asset creation and MinIO upload happen inside a loop (e.g., iterating over ads), the `create_task` call goes inside the loop, once per asset.
    - The auto-fill task is fire-and-forget — it handles its own exceptions internally. No try/except wrapper needed at the call site.

    **Per-service specifics:**

    1. **meta_sync.py**: Look for the asset download section (around the `upload_file` calls near line 820-1050). There are typically two upload points — one for image assets and one for video assets. Add the `create_task` call after EACH upload point, but only once per asset (not once per thumbnail). Find where `asset.id` and `asset.organization_id` are available after the DB commit.

    2. **tiktok_sync.py**: Same pattern — find the MinIO upload point and add the hook after the asset record is committed.

    3. **google_ads_sync.py**: Same pattern.

    4. **dv360_sync.py**: Same pattern.

    **Testing note:** These changes are integration-level wiring. The unit test for non-blocking behavior is already covered in test_ai_autofill.py from Plan 01. The sync services themselves are not unit-tested in this project (they require live platform API mocks).
