**auth.service.ts — full rewrite of token management:**

1. Remove `ACCESS_KEY` and `REFRESH_KEY` localStorage key constants (e.g., `private readonly ACCESS_KEY = 'bs_access'` or similar).

2. Remove these methods entirely: `storeTokens()`, `clearTokens()`, `getRefreshToken()`, and any `localStorage.getItem`/`setItem`/`removeItem` calls.

3. Add a private BehaviorSubject for the access token:
```typescript
private accessToken$ = new BehaviorSubject<string | null>(null);
```

4. Rewrite `getAccessToken()`:
```typescript
getAccessToken(): string | null {
  return this.accessToken$.value;
}
```

5. Add a boolean accessor:
```typescript
get isAuthenticated(): boolean {
  return this.accessToken$.value !== null;
}
```

6. Rewrite `login()`:
```typescript
login(email: string, password: string): Observable<{access_token: string; token_type: string}> {
  return this.http.post<{access_token: string; token_type: string}>(
    `${environment.apiUrl}/auth/login`,
    { email, password },
    { withCredentials: true }  // browser stores the httpOnly Set-Cookie
  ).pipe(
    tap(res => this.accessToken$.next(res.access_token))
  );
}
```
The response no longer contains `refresh_token` (backend Plan 03 removed it). `withCredentials: true` tells the browser to accept and store the `Set-Cookie` header from the response.

7. Rewrite `refreshTokens()`:
```typescript
refreshTokens(): Observable<{access_token: string; token_type: string}> {
  return this.http.post<{access_token: string; token_type: string}>(
    `${environment.apiUrl}/auth/refresh`,
    {},  // empty body — refresh token is in the httpOnly cookie
    { withCredentials: true }  // sends httpOnly cookie automatically
  ).pipe(
    tap(res => this.accessToken$.next(res.access_token))
  );
}
```

8. Rewrite `logout()`:
```typescript
logout(): void {
  this.http.post(
    `${environment.apiUrl}/auth/logout`,
    {},
    { withCredentials: true }
  ).subscribe();
  this.accessToken$.next(null);
  this.user$.next(null);
  this.router.navigate(['/auth/login']);
}
```

**auth.guard.ts — add silent refresh before redirect (QUAL-04 fix):**

The current guard checks `auth.isAuthenticated` which is now memory-based (no localStorage). On page refresh, `isAuthenticated` is always false. The guard must attempt a silent refresh before redirecting to login:

```typescript
import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { AuthService } from '@core/services/auth.service';
import { map, catchError, of } from 'rxjs';

export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);

  // If already authenticated in memory, allow immediately
  if (auth.isAuthenticated) {
    return true;
  }

  // Attempt silent refresh (browser sends httpOnly cookie automatically)
  return auth.refreshTokens().pipe(
    map(() => true),
    catchError(() => of(router.createUrlTree(['/auth/login'])))
  );
};
```

Read the existing guard file first to understand its current structure (it may be a class-based guard or functional guard). Adapt the pattern to match the existing style. If class-based, convert to functional `CanActivateFn` or keep class-based and add the refresh logic in `canActivate()`.

**auth.interceptor.ts — ensure cookie support on refresh retry:**

Read the existing interceptor. It likely already calls `auth.refreshTokens()` on 401. After Task 1, `refreshTokens()` uses `withCredentials: true` internally, so the interceptor itself may not need changes. However, verify:

1. The interceptor does NOT manually attach a refresh token to the retry request body (old behavior). If it does, remove that.
2. The interceptor calls `auth.refreshTokens()` and then replays the original request with the new access token from `auth.getAccessToken()`.
3. If the refresh itself fails (returns 401), the interceptor calls `auth.logout()` and does NOT infinite-loop.
4. Original requests that need the access token should get it from `auth.getAccessToken()` and set `Authorization: Bearer ${token}` header.

Ensure the interceptor does NOT add `withCredentials: true` to all requests (only auth service's refresh/login/logout calls need it). Regular API calls send `Authorization: Bearer` header instead.

**app.config.ts (or app.module.ts) — optional APP_INITIALIZER:**

The RESEARCH.md suggests adding an `APP_INITIALIZER` to attempt silent refresh on app startup. This is an alternative to the guard-based approach. Since the guard already handles it, the APP_INITIALIZER is optional but provides a better UX (user sees loading state instead of a flash of login page).

If the app uses standalone bootstrapping (check `main.ts` or `app.config.ts`):
```typescript
import { APP_INITIALIZER } from '@angular/core';
import { AuthService } from '@core/services/auth.service';
import { catchError, of } from 'rxjs';

function initAuth(authService: AuthService) {
  return () => authService.refreshTokens().pipe(
    catchError(() => of(null))  // silent failure — guard handles redirect
  );
}

// Add to providers array:
{
  provide: APP_INITIALIZER,
  useFactory: initAuth,
  deps: [AuthService],
  multi: true,
}
```

If the app uses a module-based setup (AppModule), add the provider there instead.

If adding APP_INITIALIZER is complex due to the app structure, skip it — the guard-based approach is sufficient. Document the decision.
